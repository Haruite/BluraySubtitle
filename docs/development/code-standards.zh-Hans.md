# 代码修改规范

[English](code-standards.md)

## 1. 权威来源与适用范围

本规范是所有向本仓库提交 PR 或代码修改的贡献者必须遵守的强制要求。

规则权威顺序如下：

1. 本规范；
2. `README.md` 和 `README.zh-Hans.md` 描述的产品行为；
3. 不与以上内容冲突的实现细节。

代码可以包含比 README 更详细的实现，但不能与 README 冲突。确认需要保留的实现行为必须同步写入中英文 README。

作者确定新规则后，必须在依赖该规则的代码修改之前或同一次修改中同步更新本文件及英文版本。重构或重大改动还必须同步更新中英文[重构历史](../refactoring/refactoring-history.zh-Hans.md)；一般修改不需要增加重构历史记录。

## 2. 简单与一致

- 使用能够正确解决问题的最简单实现。
- 在当前修改范围内删除重复、冲突、不可达和不必要的逻辑。
- 不为产品并不需要的情况增加推测性校验或抽象。
- 避免任何与目的无关的不必要校验。优先使用简单的结构或字面量检查，只有格式本身属于需求时才使用严格的正则表达式匹配。
- 在拥有该事实的边界校验一次，不要让每一层重复相同校验。
- 真正共用的行为只保留一份实现；不能仅因为代码看起来相似，就强迫无关工作流共用抽象。
- 一个函数完成一个完整操作。不要把直接的操作拆成大量只转发参数或只有一行的短函数。
- 在不制造无关巨型函数的前提下，尽量减少函数数量。
- 名称应表达领域值或操作。除惯用且范围很小的索引外，不使用无意义临时名称。
- 导入、类型标注、异常处理和格式应与所在模块保持一致。
- 除非仓库内部调用方或已确认需求需要，否则不保留兼容包装层或 facade API。
- 宽泛异常处理不能隐藏无效配置或执行失败。只有在明确的 GUI/Worker 边界或尽力清理边界，并且失败仍得到恰当处理时才可使用。

## 3. 源码语言、注释与命名

- Python 源码字符串一律使用英文；`I18N_ZH_TO_EN` 中作为目录键使用的中文是特定例外。
- 代码注释和 docstring 一律使用英文。
- 对关键领域规则、不直观的计算、所有权边界和有意保留的例外补充注释。
- 不添加只是重复显而易见代码含义的注释。
- 变量、函数、类和字段使用有实际含义的名称。

## 4. GUI 是执行契约

- 当前 GUI 可见状态具有最高权威：已选行、顺序、路径、名称、语言、模式、命令、章节范围、编码器及其他选项都必须准确应用。
- 任务启动时一次性捕获 GUI 状态，并尽可能通过一个显式 Request 传递。
- Worker 或 Service 不能把显式 GUI 值静默替换为旧快照、全局值、默认值、重新生成值或推测值。
- 不能静默跳过 GUI 中已选择的行或选项。无法执行时必须报错。
- 只有用户没有提供显式值时才能自动推断。
- 除非 GUI 明确说明其他排序规则，执行顺序必须与启动时捕获的可见表格顺序一致。
- Worker 拥有已经捕获的 Request，启动后不能再读取实时 GUI 控件。
- Service 只消费普通 Python 数据，不能读取或重新解释 Qt 表格和控件。
- 长时间任务必须离开 GUI 线程执行，并保持进度、取消、成功和错误处理一致。

## 5. 预检查与失败处理

- 可预知且可操作的问题尽可能提前检查，优先在 Worker 启动前检查。
- 预检查应集中于 Request 中已经确定的事实：必需路径、已选输入、所需工具、无效区间、不完整行映射、命令结构、输出路径和可预知冲突。
- 不重复执行昂贵的媒体探测，也不在没有已确认规则时增加会拒绝有效输入的苛刻校验。
- 执行时才发现的媒体相关失败必须指出受影响的源文件或行，并明确报错。
- 计划输出已存在时默认报错；只有已确认的产品规则明确将工作流定义为可恢复时才可例外。已有输出不能被覆盖、改名，也不能被复用为其他结果。
- 可恢复工作流必须记录哪些输出类型可视为已完成，并在任务进度中报告每一个跳过的输出。
- 能在执行前确定输出路径时，必须在第一次写入前计算并检查完整输出集合。
- 能在执行前完成映射规划时，必须先完成规划，再修改源文件或创建最终输出。
- 外部命令必须检查返回状态。只有已经记录或验证的工具警告返回码才可以视为成功。
- 外部工具优先使用参数列表和 `shell=False`；只有确实需要 shell 语法时才能使用 shell。
- 清理操作只能删除当前任务创建的临时文件或不完整产物，不能删除用户原本存在的文件。

## 6. 分层职责

- GUI/配置层：一次性读取当前控件，规范化显式值，执行确定性预检查并创建完整 Request。
- Worker 层：拥有一个 Request、进度回调、取消状态及最终成功/错误信号。
- Service 层：根据普通数据执行工作流和领域逻辑，不查询 GUI 或隐藏全局状态。
- 领域/工具层：执行可复用的媒体计算和显式外部工具操作。
- 避免可变的模块级工作流配置。
- 不要把同一个配置既写入 Service 属性，又作为方法参数重复传递。
- 只共享领域计算和写入原语，不共享过期的工作流状态。
- `src/runtime/gui_runtime_split` 中 mixin 实现的所有方法，必须在 `gui_base.py` 中有签名一致的声明。
- `src/runtime/services_split` 中 mixin 实现的所有方法，必须在 `service_base.py` 中有签名一致的声明。

## 7. i18n 与用户可见文本

- 每个 GUI 字符串、对话框字符串、进度文本、终端信息和其他用户可见输出都必须提供英文和简体中文版本。
- 生产代码使用英文作为源字符串。
- 在同一次修改中把英文映射加入 `src/core/i18n.py:I18N_ZH_TO_EN`。
- 在合适的展示边界通过 `self.t(...)` 或 `translate_text(...)` 输出用户可见文本。
- 动态信息应翻译稳定模板，再填入具体值。
- 记录产品行为或需要保留的实现细节时，必须同时更新 `README.md` 和 `README.zh-Hans.md`。
- README 只描述当前功能、需要保留的实现行为和使用注意事项。历史行为对比、已删除行为、重构原因和未来清理计划应写入重构历史或开发文档，不能写入 README。

## 8. 文件格式

- 源码和文档使用 UTF-8。
- 所有新增或修改的非 `.sh` 文本文件必须使用 CRLF。
- shell 脚本必须使用 LF，避免 shebang 失效。
- 不引入行尾空格或错误编码。

## 9. 已确认的产品约束

- Remux 来源 Encode 支持恢复执行。已有的计划正片、SP、外挂字幕和附带文件输出视为已完成并跳过且不覆盖，其余行继续执行；当前请求内部的重复路径仍然报错。
- 原盘 DIY 保持可见并保留代码；未完成的执行逻辑不能表现为已经完成。
- 配置的轨道语言修改必须真正应用到输出。
- 原盘 Remux 和原盘 Encode 不支持通用视频转换；未来 DIY 视频转换需要单独确认设计。
- 每个已选择的主 MPLS 必须且只能对应一条非空主 Remux 命令。
- README 已记录的技术回退可以改变实现方法，但必须保持已选择轨道、语言、章节范围、名称和输出路径等显式 GUI 意图。
- 添加章节按照当前可见 MKV 顺序与所选主播放列表依次匹配，不要求外部 MKV 文件名包含 `BD_Vol_NNN`。

## 10. 测试与修改报告

- 每个修改过的工作流边界和可确定测试的已修复回归都要补充针对性自动测试。
- 一个工作流或阶段完成时执行仓库集中测试。
- 根据修改范围，至少从以下检查中执行适用项目：
  - Python 编译和导入冒烟测试；
  - 仓库单元测试；
  - `tools/check_i18n.py`；
  - `tools/check_split_contracts.py`；
  - `git diff --check`；
  - 新增和修改文件的 CRLF/LF 检查。
- 完整蓝光和 MKV 素材属于手动回归输入，不是 CI 固件。必须准确说明仍需执行哪些真实素材测试、会写入什么，以及是否必须使用可丢弃副本。
- 每批修改结束时必须报告：
  - 修改的文件和区域；
  - 删除的冗余或冲突路径；
  - 每个业务逻辑变化，包括旧行为和新行为；
  - 已执行测试及结果；
  - 仍需完成的真实素材检查；
  - README、i18n 和规范更新；属于重构或重大改动时还要报告重构历史更新。
