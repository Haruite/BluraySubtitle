## 用途

本文档长期记录 Python 重构情况，包括每个已完成阶段或工作流修改了什么、业务逻辑如何变化、执行了哪些测试，以及哪些内容被明确推迟。

Phase 1 的详细分析仍保存在 [第一阶段重构契约与配置矩阵](phase-1-contract-and-configuration-matrix.md) 中。后续所有修改还必须遵守强制性的[代码修改规范](../development/code-standards.zh-Hans.md)。

## 维护规则

每次完成重构或重大改动，都必须在同一次修改中同步更新本文档及其英文版本；一般修改不需要增加历史记录。每条记录必须包含：

- 范围与提交；
- 删除的冗余或冲突路径；
- 业务逻辑变化，包括旧行为和新行为；
- 文档与 i18n 变化；
- 自动检查及仍需使用真实素材完成的手动检查；
- 明确推迟的工作。

历史记录必须遵循作者已经记录的意图；尚未确定的行为应标记为待确认，不能为了补全文档而自行给出结论。

### 文档与 i18n

## Phase 3.2——添加章节工作流

日期：2026-07-20
提交：`107cea1`（`refactor: unify add chapters workflow`）

### 范围

将“添加章节”重构为独立的 GUI Request、后台 Worker 和纯数据 Service 路径。

### 删除的冗余或冲突路径

- 删除 GUI 线程中的同步执行。
- 删除吞掉 GUI 配置错误后回退到旧表格/配置逻辑的路径。
- 删除 Service 边界中的 Qt 表格对象。
- 删除对工作目录共享 `chapter.txt` 的依赖。
- 删除对 `completion()` 的调用，避免把“直接编辑原文件”误解为允许补全蓝光目录。

### 逻辑变化

- 请求在启动时一次性捕获当前可见 MKV 顺序、所选主播放列表、准确的源/输出路径对和直接编辑模式。
- MKV 与所选播放列表根据时长和章节点按顺序匹配，不要求文件名包含 `BD_Vol_NNN`。
- 修改任何 MKV 前先完成全部章节文档规划；所选播放列表无法覆盖全部 MKV 时，在写入前失败。
- 主播放列表、MKV 输入、所需 MKVToolNix 程序、重复路径和已有输出在可预知时于 Worker 启动前检查。
- 已存在输出明确报错。
- 每个任务使用独立临时章节文件。
- `mkvmerge`、`mkvpropedit` 不再通过 shell 命令字符串执行，失败返回码会被报告；MKVToolNix 返回码 1 仍视为带警告的成功。
- 只有单个零点章节时，直接编辑仍保持源文件不变；新文件模式仍会生成 MKV，但不会加入这个无意义章节。

### 文档与 i18n

- 同步更新中英文 README，记录顺序、播放列表匹配、直接编辑/新文件模式、预检查和输出冲突规则。
- 新增工作流、工具、映射和命令失败所需的中英文信息。
- 同步 Worker 导出以及 GUI/Service IDE 基类契约。

### 验证

- 集中执行仓库测试，共 49 项全部通过。

## Phase 3.3——原盘 Remux 工作流

日期：2026-07-22
提交：`b89f995`（`refactor(remux): rebuild main playlist workflow`）

### 范围

重构 Remux 的 GUI 请求、Worker 边界、预执行规划、主播放列表执行和最终输出映射。保留已有 fallback 算法，但每个已选择主播放列表现在都在一个确定的独立任务中执行。

### 删除的冗余或冲突路径

- 删除 Worker 中镜像 GUI 状态的大量参数和重复的 Service 配置赋值。
- 删除按原盘分组后执行的路径；旧路径在同一原盘存在多个已选择主播放列表时可能只执行第一个。
- 删除提前创建输出目录、随后再扫描目录以重新发现本任务输出的逻辑。
- 删除输出名称字符自动替换、数字冲突后缀和回退到无关原始输出文件的逻辑。
- 删除强制启用“补全蓝光目录”的逻辑。

### 逻辑变化

- 一个自有 Remux 请求一次性捕获当前 GUI 配置、已选择主播放列表、可见输出名称和语言、SP 行、轨道设置、默认音频编码、电影模式、裁剪、Dolby Vision 和目录补全选项。
- 按界面可见顺序规划主播放列表。每个已选择主播放列表必须生成且只生成一条非空命令和一个执行任务，同一原盘中的多个主播放列表也不例外。
- GUI 命令预览与执行逻辑现在共同使用同一个七项命令结果。命令生成失败时预览保持为空，并在捕获配置时被拒绝，不再合成一条缺少计划输出路径和轨道参数、但看起来可以执行的命令。
- 创建输出目录前推导所有命令理论输出和最终 GUI 输出路径，并在写入前校验配置行数、章节范围、重复路径和已存在文件。
- 最终名称严格按照界面显示值应用；没有 `.mkv` 扩展名时补充扩展名，无效的 Windows 文件名明确报错。
- 主命令或 fallback 必须生成对应任务规划的全部输出。缺少输出时任务现在会失败，不再静默跳过；失败后删除本任务新建的预期中间输出。
- 收尾处理直接使用规划输出列表，不再扫描目标目录，并使用任务独立的临时章节文件。
- 无论是否勾选目录补全，章节元数据都会直接写入本任务新生成的输出；该复选框现在只控制蓝光目录补全。
- 每个主播放列表任务都会捕获“编辑轨道”保存的语言，并在创建输出前检查 `mkvpropedit` 是否可用。主命令或任一 fallback 成功后，只对实际包含轨道应用已配置语言，修改后重新识别输出并验证；映射、命令或验证失败都会使任务失败并删除该任务新建的主输出。
- 剧集输出顺序继续与配置顺序对齐，供后续字幕、语言、SP、音频和 Dolby Vision 处理使用。
- “补全蓝光目录”严格使用捕获到的复选框值。

## Phase 3.4——原盘 Encode 主流程

日期：2026-07-22
提交：`d4adee2`（`refactor(encode): unify Blu-ray and Remux encode workflows`）

### 范围

重构 Encode 的启动、Worker、原盘暂存及原盘/Remux 两种输入共用的逐行执行路径。SP 混流算法、轨道对齐、缺轨修复、音频转换算法和 Dolby Vision 算法仍留在后续对应工作流中。

### 删除的冗余或冲突路径

- 删除 GUI 中原盘/Remux 两套重复的启动分支、线程清理和信号连接。
- 删除 Worker 镜像 GUI 状态的大量参数，改为一个不可变 Encode 请求。
- 删除 `EncodeMkvFolderWorker` 以及 Service 内同步嵌套调用 Worker 的路径。执行错误现在会直接传递给唯一的 GUI Worker，不再由内部 Worker 转成信号后被外层继续当作成功。
- 删除字幕、输出名称、语言、VPy 路径、配置和 SP 条目的并行数组；每个可见行直接拥有相关值和准确输出路径。
- 删除 Remux 后扫描目录以重新发现暂存正片输出的逻辑。
- 删除 Encode 对隐藏“补全蓝光目录”复选框的读取、强制启用状态以及 `completion()` 调用。
- 删除旧有的已有输出静默跳过、复制失败尽力忽略及运行时重新生成缺失显式 VPy 文件的行为；Remux 来源恢复执行改为遵循一条明确记录的统一规则。

### 逻辑变化

- 一个请求在 Worker 启动前捕获输入模式、正片/SP 行顺序、准确输出、字幕与语言、VPy 路径、全部压制参数、裁剪与 Dolby Vision 选项和轨道设置。
